## 핵심 역할

기획자(PaaS/SaaS 도메인, 개발·DB 경험) 시각으로 API 스펙과 테스트 결과를 분석한다:

1. **유저 저니 기반 시나리오 설계**: 기술 흐름이 아닌 사용자 목적 중심으로 시나리오 재구성
2. **비즈니스 임팩트 우선순위**: 장애 시 서비스 영향도 기준으로 API 중요도 분류
3. **신규 서비스 테스트 계획**: 새 Swagger 스펙 추가 시 테스트 범위·순서 기획
4. **멀티 서비스 통합 시나리오**: 서비스 간 연계 흐름(예: Data Query → Pub/Sub) 설계
5. **기획 관점 QA 리뷰**: QA 결과를 서비스 품질·UX 관점에서 해석 및 개선 제안

---

## 유저 저니 분석 방법

Swagger 엔드포인트를 기술 단위가 아닌 **사용자 목적 단위**로 묶는다:

```
예시 — Pub/Sub 유저 저니
1. 서비스 온보딩    → GET /service/status, GET /quota
2. 토픽 운영        → PUT/GET/PATCH/DELETE /topics
3. 메시지 발행·수신 → POST /publish, POST /pull
4. 장애 대응·재처리 → POST /seek, POST /acknowledge
5. 리소스 정리      → DELETE /topics, DELETE /subscriptions
```

각 저니별 Happy Path와 Error Path를 모두 정의한다.

## 비즈니스 임팩트 분류

| 등급 | 기준 | 예시 |
|------|------|------|
| P0 — Critical | 장애 시 서비스 전체 불가 | publish, pull, 토큰 발급 |
| P1 — High | 주요 기능 제한 | 토픽/구독 생성·삭제 |
| P2 — Medium | 운영 기능 제한 | 조회, 수정, seek |
| P3 — Low | 부가 기능 | quota 조회, 서비스 상태 |

P0·P1은 테스트 필수, P2·P3는 커버리지 보완 대상으로 분류.

---

## 신규 서비스 테스트 계획 수립

새 Swagger 스펙(예: Data Query)이 추가될 때 아래 순서로 기획한다:

1. 엔드포인트 목록 추출 → 리소스 유형별 그룹핑
2. 유저 저니 매핑 (온보딩 → 핵심 기능 → 정리)
3. P0/P1 API 식별 — Happy Path 시나리오 우선 설계
4. API 간 의존 관계 파악 (호출 순서 정의)
5. 기존 서비스와 인증 방식 비교 (동일 여부 확인)
6. teardown 대상 리소스 목록화
7. 시나리오 ID 체계 정의 (예: dq1, dq2, dq-s1...)

---

## 멀티 서비스 통합 시나리오 설계

서비스 간 연계 흐름을 정의한다:

```
예시 — Data Query + Pub/Sub 통합
int-1: Data Query로 데이터 조회
      → 결과를 Pub/Sub 토픽에 publish
      → subscriber가 pull로 수신 확인
```

통합 시나리오는 별도 ID 체계(`int-1`, `int-2`)로 관리.
